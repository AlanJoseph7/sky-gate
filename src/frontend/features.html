// CAPABILITIES

Built for production

Sub-second inference, live API integration, modular architecture, and structured logging — designed to run zero-cost on CPU hardware with no cloud dependencies.

// SYSTEM FEATURES
🛰️

OpenSky + adsb.lol Integration

Direct REST API pulls over Indian airspace bounding box. Dual-source feeding — OpenSky for authenticated deep history, adsb.lol for open real-time feed. Configurable polling interval.

⚡

Sub-second Inference

All three pipeline layers run sequentially in under 400ms per batch, enabling real-time detection without queuing lag. No GPU required — optimised for CPU-only execution.

📊

Feature Engineering

Velocity deltas, heading consistency, barometric vs. geometric altitude divergence, and temporal sequence embedding. Per-aircraft history buffer resolves NaN issues from single-snapshot data.

🔔

Severity Triage

Three-tier severity — INFO, WARN, CRITICAL — with per-anomaly drill-down and confidence scores. Emergency squawks (7500/7600/7700) trigger immediate CRITICAL classification.

🗂️

Structured Logging

All detections persisted to structured JSON logs with ICAO24, callsign, detection layer, score, and UTC timestamp. SSE-powered live feed streams logs directly to the dashboard.

🧩

Modular Architecture

Each pipeline layer is independently swappable. Drop in XGBoost, VAE, or custom rules via a clean detector interface. Each module is runnable standalone via python {module}.py.

🔄

Background Retraining

Scheduled retraining loop ingests new live data and updates model artifacts on disk automatically. Dynamic F1 weights recalculate post-retrain, keeping the ensemble self-correcting.

📡

Live Dashboard

FastAPI-powered dashboard with SSE log feed, anomaly alert table, leaderboard, and real-time stats. Dark radar aesthetic with full light/dark theme support.

🔁

Reproducible Pipeline

Seeded RNG throughout (RANDOM_SEED = 42). Saved model artifacts mean training only runs once. Each step independently repeatable.

// KEY DESIGN DECISIONS
Why use both live and synthetic data?▾

Live ADS-B data from OpenSky and adsb.lol provides real-world normal flight behaviour over Indian airspace, ensuring the models learn genuine traffic patterns. Synthetic anomalies are then injected into this real data for supervised training and evaluation, since real labelled attack data is not publicly available.

Why a feedforward autoencoder instead of LSTM?▾

The LSTM autoencoder was initially used but produced an anomaly/normal reconstruction ratio of ~0.97x — effectively random. The feedforward autoencoder evaluates each row independently, making anomalous feature combinations directly visible as reconstruction error spikes. Final ratio: 1.55x.

Why dynamic F1 weights in the ensemble?▾

Hardcoded weights (e.g. 0.4/0.3/0.3) don't adapt to actual detector performance. Dynamic weights mean the ensemble automatically favours whichever detector is strongest on the current dataset — if the autoencoder degrades, it automatically loses voting power.

Why clip outliers before MinMaxScaler?▾

Anomaly values (e.g. teleport: 500km/step) dominated the MinMaxScaler range, collapsing all normal variation to near zero. Clipping to 4σ of normal data preserves normal feature variation while keeping anomaly signal intact.
